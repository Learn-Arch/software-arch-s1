## Задача 1: 
#### Основные функции Mesto:
1. Загрузка фотографий.
2. Удаление фотографий.
3. Сбор и учёт лайков под фото.
4. Создание профиля и его редактирование.

#### Предлагаемая декомпозиция
Исходя из списка функций портала, предлагается декомпозировать монолитный фронтенд на следующие модули, используя стратегию вертикальной нарезки:

1. **Main (главное приложение)**:
  - Отвечает за:
    1. Роутинг между фронтендами.
    2. Композицию микрофронтендов профиля и карточек.
    3. Глобальный контекст приложения.

2. **Auth (аутентификация)**:
  - Обработка процессов входа и регистрации новых пользователей.

3. **Profile (профиль пользователя)**:
  - Обработка всех функций, связанных с профилем пользователя, таких как редактирование профиля и загрузка фото.

4. **Cards (карточки)**:
  - Отвечает за процессы загрузки, отображения, редактирования карточек, а также обработку лайков.

#### Минимизация зависимостей
Каждый модуль проектируется таким образом, чтобы минимизировать зависимости между ними. Это можно достичь за счёт использования паттерна BFF (Backend for Frontend), который отвечает за преобразование данных и их передачу фронтенду.

### Выбор инструмента для создания микрофронтендов

Для создания микрофронтендов предлагается использовать **Webpack Module Federation**. Этот инструмент позволяет разделить код на независимые модули, которые загружаются и используются по мере необходимости.

Преимущества **Webpack Module Federation** для Mesto:
1. Приложение построено на едином технологическом стеке (React).
2. Возможность масштабирования: в случае роста количества модулей возможно перейти на **Single SPA**, чтобы использовать разные фреймворки.
3. Упрощённая передача данных между микрофронтендами.
4. Независимость модулей за счёт использования паттерна BFF.

## Описание платформы и её основных бизнес-функций

Платформа представляет собой сервис для продажи и приобретения услуг клиентами, которым можно присвоить две ключевые роли:
1. Заказчики — клиенты сервиса, заинтересованные в поиске подрядчика для оказания определённых услуг или поставке товаров.
2. Подрядчики — клиенты сервиса, заинтересованные в продаже своих товаров и услуг.

Также сервис предусматривает две дополнительные роли пользователей, отвечающих за поддержку и администрирование площадки:
1. Администратор.
2. Специалист поддержки.

### Продажа услуг через сервис осуществляется двумя основными способами:
1. Поиск и покупка услуги заказчиком путем размещения заказа с последующим заключением сделки на платформе.
2. Выбор лучшей цены заказчиком посредством аукциона.

Помимо соединения интересов заказчиков и подрядчиков, система предоставляет услугу обеспечения безопасной сделки для обеих сторон. Оплата осуществляется через внешние платёжные сервисы, а безопасность обеспечивается возможностью возврата средств заказчику в случае удовлетворения апелляции, если товар или услуга не были предоставлены подрядчиком.

За успешное завершение сделки сервис получает комиссию, что является основным источником прибыли и одной из наиболее ценных функций для бизнеса.

* Слабая связанность — Каждый микросервис имеет свою область ответственности и минимизирует зависимость от других.
* Разграничение контекстов — У каждого микросервиса свой отдельный домен, соответствующий бизнес-логике.

## Описание потоков данных

1. **Инициировать платёж** — торговая площадка инициирует платёж, отправляя необходимые данные (ID пользователя, ID заказа, сумма заказа) в платёжный сервис.
2. **Предоставить статус платежа** — платёжный сервис принимает запрос на проведение оплаты, подтверждает возможность проведения платежа со счёта клиента, назначает идентификатор для этой транзакции и устанавливает статус.
3. **Подтверждение платежа** — торговая площадка уведомляет платёжный сервис о подтверждении операции, отправляя ID транзакции.
4. **Запросить статус платежа** — торговая площадка отправляет запрос в платёжный сервис для получения текущего статуса обработки транзакции по ID.
5. **Подтверждение отмены платежа** (альтернативный сценарий) — платёжный сервис уведомляет систему об успешной отмене платежа, предоставляя статус отменённой транзакции.
6. **Инициировать отправку уведомления** — в случае регистрации пользователя, изменения состояния заказа или аукциона соответствующий микросервис отправляет запрос на отправку уведомления микросервису “Уведомления”.
7. **Отправка уведомления пользователю** — микросервис “Уведомления” отправляет уведомления пользователю.
8. **Запрос данных об активности пользователей** — микросервис “Отчёты” запрашивает данные об активности пользователей у микросервиса “Пользователи”.
9. **Предоставление данных активности пользователей** — в ответ на запрос отчёта об активности пользователей микросервис “Пользователи” агрегирует запрашиваемые данные и отправляет их микросервису “Отчёты”.
10. **Запрос данных о продажах** — микросервис “Отчёты” запрашивает данные о заказах для создания отчёта о продажах у микросервиса “Заказы”.
11. **Предоставление данных о продажах** — в ответ на запрос отчёта о продажах микросервис “Заказы” агрегирует запрашиваемые данные и отправляет их микросервису “Отчёты”.
12. **Предоставление отчёта** — в ответ на запрос администратора через портал управления торговыми площадками микросервис “Отчёты” предоставляет порталу запрашиваемые данные.
13. **Запрос возмещения суммы заказа** — в случае положительного решения по апелляции по невыполненному заказу микросервис “Техподдержка” отправляет запрос на возмещение суммы заказа микросервису “Платежи”.
14. **Подтверждение заказа на исполнение аукциона** — для подтверждения заказа микросервис “Заказы” запрашивает информацию о товарах и услугах (актуальную стоимость услуги или наличие товаров на торговой площадке) у микросервиса “Аукционы”.
15. **Запрос данных о составе аукциона** — микросервис “Аукционы” запрашивает информацию о товарах и услугах (актуальную стоимость услуги или наличие товаров), которые входят в состав аукциона, у микросервиса “Торговые площадки”, где размещены эти данные.
16. **Предоставление данных о составе аукциона** — в ответ на запрос данных о товарах и услугах микросервис “Торговая площадка” предоставляет эти данные микросервису “Аукционы”.
17. **Предоставление данных об аукционе** — в ответ на запрос данных о составе аукциона микросервис “Аукционы” предоставляет эти данные микросервису “Заказы” для дальнейшего подтверждения заказа.
18. **Запрос данных о товаре или услуге** — для подтверждения заказа, размещённого напрямую на торговой площадке пользователя, микросервис “Заказы” запрашивает данные о товарах и услугах у микросервиса “Торговые площадки”.
19. **Предоставление данных о товарах и услугах** — в ответ на запрос данных о товарах и услугах микросервис “Торговые площадки” предоставляет эти данные.
20. **Инициация платежа** — после подтверждения заказа микросервис “Заказы” инициирует платёж, отправляя все необходимые данные в микросервис “Платежи”.
21. **Предоставление статуса платежа** — после проведения платежа микросервис “Платежи” возвращает статус платежа.

## Приоритизация микросервисов

Для осуществления процесса миграции с монолитной системы на микросервисную архитектуру, используя паттерн Strangler Fig, список выделенных микросервисов необходимо приоритизировать. Ввиду того, что недостаточно данных для технического анализа и анализа производительности, я предложил бы изымать функции из монолита и создавать микросервисы, руководствуясь влиянием на бизнес и клиентов, в следующем порядке:
1. **Заказы** — сервис заказов (а также связанный с ним сервис платежей) обеспечивает бизнесу основной источник прибыли. С точки зрения бизнес-влияния их необходимо выделять в микросервис в первую очередь.
2. **Платежи** — в зависимости от данных технического анализа и анализа производительности сервис платежей также может занять первое место в приоритетах.
3. **Пользователи** — сервис, осуществляющий авторизацию и регистрацию новых пользователей, являющийся основной точкой входа на платформу, без сомнения должен быть в приоритете с точки зрения влияния на клиентов.
4. **Торговые площадки** — отказоустойчивость и возможность предоставлять данные о товарах, услугах и аукционах без лишнего времени ожидания положительно скажутся на пользовательском опыте.
5. **Аукционы** — сервис аукционов агрегирует данные о товарах и услугах торговых площадок, добавляя дополнительный функционал. Если проведение аукционов не является ключевой функцией или конкурентным преимуществом текущей платформы, его можно выделить в микросервис после торговых площадок.
6. **Техподдержка** — грамотная и бесперебойная работа специалистов технической поддержки, которые также выполняют ключевую функцию обработки апелляций и возврата средств, может стать важным конкурентным преимуществом.
7. **Уведомления** — надёжность сервиса уведомлений пользователей обычно оказывает наименьшее влияние на пользователей в подобных системах.
8. **Отчёты** — данные для составления отчётности будут надёжно храниться и агрегироваться сервисами “Заказы” и “Пользователи”, оставаясь доступными для анализа. Поэтому удобный для владельцев бизнеса сервис быстрой генерации отчётов можно временно отложить.
